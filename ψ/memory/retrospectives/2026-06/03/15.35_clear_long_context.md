## Key Findings

### 🔴 Core Issue: Context Bloat (900+ Steps)
- **Root cause**: The session grew excessively long (900+ steps, 1MB+ JSONL transcript). In the Gemini/Antigravity CLI ecosystem, each turn or tool call sends the entire conversation history. This caused token consumption to balloon to hundreds of thousands of tokens per single tool invocation.
- **Impact**: Hourly rate limits were hit rapidly, forcing the model to fail and require manual model switching.
- **Resolution**: Promptly run `/rrr` and wrap up the session to start a clean one, flushing the context window.

## Honest Feedback

**Friction 1**: ระบบ Discord Inbound Relay (`no6-discord-relay.py`) คอยดึงข้อความใหม่และส่งผ่าน `maw hey 06-gemini` ซึ่งจะทำการยัดข้อความเข้าเซสชั่นเดิมไปเรื่อยๆ ทำให้ประวัติสะสมยาวนานจนเกินขนาดที่เหมาะสมโดยไม่มีการตัดรอบเซสชั่นอัตโนมัติ

**Friction 2**: การขาดสัญญาณเตือนหรือตัววัดขนาด Context Window ภายในตัวเอเจนต์ ทำให้เอเจนต์ไม่รู้ตัวเลยว่าขณะนี้กำลังใช้ทรัพยากรหลักล้านโทเคนในการตอบคำถามเพียงแค่คำเดียว จนกระทั่งฝั่งผู้ใช้สังเกตเห็นอัตราการเบิร์นโทเคนและเข้ามาเตือน

**Friction 3**: การหาค่าโควต้าของ Gemini API เป็นเรื่องยากเนื่องจากตัว API ดิบไม่ส่ง headers โควต้ากลับมาในแบบมาตรฐาน ทำให้เอเจนต์ต้องลองรันสคริปต์และหาข้อมูลในหลายช่องทาง ซึ่งยิ่งเป็นการเรียกใช้ทูลและส่งบริบทประวัติซ้ำๆ ทำให้ใช้โทเคนฟุ่มเฟือยมากขึ้นในจังหวะที่โควต้ากำลังจะหมด

## Lessons Learned

1. **Keep sessions short and clean** — การดึงเซสชั่นยาวเกินไปสร้างภาระโหลดและเบิร์นโควต้าอย่างมหาศาล ควรทำ Retrospective และขึ้นเซสชั่นใหม่เมื่อขั้นตอนเริ่มยาวหรือจำนวนสเต็ปเกิน 50-100 สเต็ป
2. **Context scale grows quadratically on tool loops** — ยิ่งเรียกใช้ทูลเยอะในเซสชั่นยาว โทเคนยิ่งคูณสองคูณสาม ต้องระวังเป็นพิเศษ
3. **Proactive flush is better than user warning** — อย่ารอให้ยูสเซอร์หรือ API แจ้งเตือนลิมิต ควรตรวจเช็คและเสนอตัวทำ `/rrr` เองล่วงหน้า
